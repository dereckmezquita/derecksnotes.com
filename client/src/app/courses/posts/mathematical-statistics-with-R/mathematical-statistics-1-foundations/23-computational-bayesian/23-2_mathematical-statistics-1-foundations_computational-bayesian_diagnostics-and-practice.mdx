# Chapter 23 -- Computational Bayesian Methods

## Part 2: Diagnostics and Practice

In Part 1 we built MCMC algorithms from scratch. But how do we know the chain has converged? A chain that has not converged gives misleading results -- the posterior summaries are wrong. In this part we develop diagnostic tools, learn to recognise good and bad convergence, and then graduate to production-quality Bayesian software with `rstanarm` and `brms`.

---

## 23.5 Convergence Diagnostics

### 23.5.1 The Fundamental Problem

MCMC theory guarantees convergence **in the limit**. In practice we run a finite chain and must decide: has it converged *enough*? There is no definitive test, but several diagnostics help.

### 23.5.2 Trace Plots

The simplest diagnostic: plot the chain against iteration. A well-mixed chain looks like a "fat, hairy caterpillar" -- stationary, with no trends or long excursions.


``` r
set.seed(42)

log_posterior <- function(mu, y_bar, n, sigma) {
  -n / (2 * sigma^2) * (mu - y_bar)^2 + dt(mu, df = 1, log = TRUE)
}

mh_chain <- function(n_iter, sigma_prop, start, y_bar, n, sigma) {
  chain <- numeric(n_iter)
  chain[1] <- start
  for (t in 2:n_iter) {
    proposal <- rnorm(1, chain[t - 1], sigma_prop)
    log_alpha <- log_posterior(proposal, y_bar, n, sigma) -
                 log_posterior(chain[t - 1], y_bar, n, sigma)
    if (log(runif(1)) < log_alpha) chain[t] <- proposal
    else chain[t] <- chain[t - 1]
  }
  chain
}

y_obs <- rnorm(30, 5, 2)
good_chain <- mh_chain(5000, 1.5, 0, mean(y_obs), 30, 2)
bad_chain <- mh_chain(5000, 0.05, -10, mean(y_obs), 30, 2)

trace_dt <- data.table$data.table(
  iteration = rep(1:5000, 2),
  mu = c(good_chain, bad_chain),
  quality = rep(c("Good mixing (sigma=1.5)", "Poor mixing (sigma=0.05, bad start)"), each = 5000)
)

ggplot2$ggplot(trace_dt, ggplot2$aes(x = iteration, y = mu)) +
  ggplot2$geom_line(alpha = 0.5, colour = "#0072B2") +
  ggplot2$geom_hline(yintercept = mean(y_obs), linetype = "dashed", colour = "#D55E00") +
  ggplot2$facet_wrap(~ quality, ncol = 1, scales = "free_y") +
  ggplot2$labs(x = "Iteration", y = expression(mu), title = "Trace plots: good vs poor mixing") +
  ggplot2$theme_minimal(base_size = 12)
```

### 23.5.3 The Gelman-Rubin Statistic ($\hat{R}$)

Run $M$ chains from **dispersed starting points**. If they have all converged to the same distribution, the between-chain variance $B$ should be small relative to the within-chain variance $W$.

$$\hat{R} = \sqrt{\frac{\hat{V}}{W}}$$

where $\hat{V} = \frac{n-1}{n} W + \frac{1}{n} B$ is the pooled variance estimate.

- $\hat{R} \approx 1$ indicates convergence.
- $\hat{R} > 1.1$ suggests the chains have not mixed.
- Modern recommendations use $\hat{R} < 1.01$.


``` r
set.seed(42)
n_chains <- 4
n_iter <- 5000
starts <- c(-10, 0, 10, 20)

chains <- lapply(starts, function(s) {
  mh_chain(n_iter, 1.5, s, mean(y_obs), 30, 2)
})

chains_mcmc <- coda::mcmc.list(lapply(chains, function(ch) coda::mcmc(ch[1001:5000])))
gr <- coda::gelman.diag(chains_mcmc)
sprintf("Gelman-Rubin R-hat: %.4f", gr$psrf[1])
```


### 23.5.4 Effective Sample Size (ESS)

Recall from Part 1 that ESS accounts for autocorrelation. As a rule of thumb:

- **ESS > 400** per parameter for reliable posterior means.
- **ESS > 1000** for reliable tail probabilities and credible intervals.


``` r
ess_vals <- coda::effectiveSize(chains_mcmc)
sprintf("Effective sample size: %.0f (from %d post-burn-in samples per chain, %d chains)",
        ess_vals, 4000, 4)
```

## 23.6 Diagnostic Summary: Good vs Bad Convergence

| Diagnostic | Good convergence | Bad convergence |
|:-----------|:----------------|:----------------|
| **Trace plot** | Fat hairy caterpillar | Trends, stuck regions, drift |
| **$\hat{R}$** | < 1.01 | > 1.1 |
| **ESS** | > 400 (means), > 1000 (tails) | < 100 |
| **Autocorrelation** | Drops to ~0 by lag 20 | Persists beyond lag 100 |
| **Density plot** | Smooth, unimodal | Jagged, multimodal artefacts |
| **Multiple chains** | Overlapping traces | Chains in different regions |

## 23.7 Introduction to Stan via `rstanarm`

Writing MCMC from scratch is instructive but impractical for real problems. **Stan** is a probabilistic programming language that implements Hamiltonian Monte Carlo (HMC) -- a much more efficient sampler than random walk MH. The R packages `rstanarm` and `brms` provide formula-based interfaces.

## 23.9 Posterior Predictive Checks

A fitted model should be able to generate data that **look like** the observed data. We draw replicated datasets $\mathbf{y}^{\text{rep}}$ from the posterior predictive distribution and compare them to $\mathbf{y}$.


``` r
y_rep <- rstanarm::posterior_predict(fit_bayes, draws = 100)

ppc_dt <- data.table$data.table(
  value = as.vector(t(y_rep)),
  draw = rep(1:100, each = ncol(y_rep)),
  type = "Replicated"
)
obs_dt <- data.table$data.table(
  value = as.integer(birthwt$low == "Low"),
  draw = 0,
  type = "Observed"
)

rep_means <- tapply(ppc_dt$value, ppc_dt$draw, mean)
obs_mean <- mean(obs_dt$value)

ppc_means <- data.table$data.table(
  mean_val = c(rep_means, obs_mean),
  type = c(rep("Replicated", length(rep_means)), "Observed")
)

ggplot2$ggplot(ppc_means, ggplot2$aes(x = mean_val, fill = type)) +
  ggplot2$geom_histogram(
    ggplot2$aes(y = ggplot2::after_stat(density)),
    bins = 30, alpha = 0.7, colour = "white", position = "identity"
  ) +
  ggplot2$geom_vline(xintercept = obs_mean, colour = "#D55E00", linewidth = 1.2) +
  ggplot2$scale_fill_manual(values = c("#D55E00", "#0072B2")) +
  ggplot2$labs(
    x = "Mean of low birthweight indicator",
    y = "Density",
    title = "Posterior predictive check: observed mean vs replicated means",
    fill = NULL
  ) +
  ggplot2$theme_minimal(base_size = 13) +
  ggplot2$theme(legend.position = "top")
```

## 23.11 Hamiltonian Monte Carlo (Conceptual)

Stan uses **Hamiltonian Monte Carlo (HMC)**, which augments the parameter space with "momentum" variables and simulates Hamiltonian dynamics. The key advantages:

1. **Proposals are distant** from the current state but still have high acceptance probability.
2. **Scales well** to hundreds or thousands of parameters.
3. **Low autocorrelation** -- ESS per iteration is much higher than random walk MH.

The cost is that HMC requires gradients of the log-posterior, which Stan computes via automatic differentiation. You do not need to derive gradients by hand.

- **Diagnostics build trust.** "We checked that our algorithm converged using four independent runs that all agree. The diagnostic statistics confirm reliability."
- **Software matters.** "We used Stan, a state-of-the-art Bayesian modelling platform trusted by researchers worldwide."
- **Posterior predictive checks validate the model.** "We asked the model to generate fake data and checked that it looks like the real data. This confirms the model is reasonable."
- **Compare with familiar results.** "The Bayesian and classical analyses agree closely, giving us extra confidence. The Bayesian version additionally lets us say there is a 97% probability that smoking reduces birthweight."

## Quick Reference

| Concept | Details |
|:--------|:--------|
| **Trace plot** | Visual check: "hairy caterpillar" = good mixing |
| **$\hat{R}$** | Gelman-Rubin statistic; want $< 1.01$ |
| **ESS** | Effective sample size; want $> 400$ for means |
| **Burn-in / warmup** | Discard early iterations; Stan calls it "warmup" |
| **Posterior predictive** | $y^{\text{rep}} \sim p(\tilde{y} \mid \mathbf{y})$; check model fit |
| **`rstanarm`** | `stan_glm()`, `stan_lm()`; pre-compiled Stan models |
| **`brms`** | `brm()`; generates Stan code; very flexible |
| **HMC** | Uses gradients for efficient proposals; Stan's engine |
| **Priors in `rstanarm`** | `prior = normal(0, 2.5)` (weakly informative default) |
| **Model comparison** | `loo::loo()` for leave-one-out cross-validation |
